# Requirements Document

## Introduction

このドキュメントでは、システムのルートアカウント作成および管理機能に関する要件を定義します。ルートアカウントは認証済みユーザーに 1 つだけ与えられる特別なアカウントで、複数のユーザープロフィール、グループ、アライアンスを管理するための基盤となります。ルートアカウントはユーザーの基本情報、ポイント管理、信頼度などの中核的な情報を保持し、システム内での活動の基点となります。

## Requirements

### Requirement 1

**User Story:** ユーザーとして、認証後にルートアカウントを作成したいので、安全かつ簡単にセットアップできる機能が必要です。

#### Acceptance Criteria

1. WHEN ユーザーが初めて認証される THEN システムはルートアカウント作成画面を表示する
2. WHEN ルートアカウントがまだ存在しない THEN システムは通常のログイン画面の代わりにルートアカウント作成画面にリダイレクトする
3. WHEN ルートアカウントが既に存在する THEN システムはルートアカウント作成画面へのアクセスを拒否する
4. WHEN ユーザーがルートアカウント作成フォームを送信する THEN システムはアカウント情報を検証し、問題がなければアカウントを作成する
5. WHEN ルートアカウントが作成される THEN システムは UUID v4 を使用して一意の ID を生成する

### Requirement 2

**User Story:** ユーザーとして、ルートアカウントに複数の認証方法を使用したいので、OAuth 認証オプションが必要です。

#### Acceptance Criteria

1. WHEN ユーザーが認証を完了する THEN システムは Supabase の auth.users テーブルから情報を取得し public.rootaccounts テーブルに保存する
2. WHEN ユーザーが複数の認証方法（Google、GitHub）を使用する THEN システムは認証数をカウントし所持ポイントの上限を増加させる
3. WHEN 複数の OAuth 認証が行われる THEN システムは Supabase が一つにまとめた認証情報を基準としてルートアカウントを管理する
4. WHEN 認証が完了する THEN システムは認証方法ごとにポイントを付与する

### Requirement 3

**User Story:** ユーザーとして、ルートアカウントの基本情報を設定したいので、プロフィール設定機能が必要です。

#### Acceptance Criteria

1. WHEN ルートアカウントが作成された後 THEN システムは基本情報入力画面に遷移する
2. WHEN ユーザーが地球 3 分割（主に活動している時間帯）を選択する THEN システムはその情報を保存する
3. WHEN ユーザーが母語を選択する THEN システムはその情報を保存する
4. WHEN ユーザーがサイトで使用する言語（日本語、英語）を選択する THEN システムはその情報を保存し、UI をその言語に切り替える
5. WHEN ユーザーが生誕世代を入力する THEN システムはその情報を保存する
6. WHEN ユーザーが基本の価値観に関する質問に回答する THEN システムはその情報を保存する
7. WHEN ユーザーがオアシス宣言に同意する THEN システムはその情報を保存する

### Requirement 4

**User Story:** ユーザーとして、ポイントシステムを利用したいので、ポイント管理機能が必要です。

#### Acceptance Criteria

1. WHEN ルートアカウントが作成される THEN システムは初期ポイントの上限を設定する
2. WHEN ユーザーが複数の認証方法を使用する THEN システムは所持 MAX ポイントを増加させる
3. WHEN ユーザーが基本の価値観に関する質問に回答する THEN システムは所持 MAX ポイントを増加させる
4. WHEN ポイントが消費される THEN システムはルートアカウントのポイント残高を更新する
5. WHEN 毎日日本時間の朝 9 時になる THEN システムは MAX ポイントの 50%分を補充する
6. WHEN ポイントがゼロもしくはマイナスになる THEN システムはポイントを消費する機能を制限する

### Requirement 5

**User Story:** ユーザーとして、複数のユーザープロフィールを管理したいので、プロフィール管理機能が必要です。

#### Acceptance Criteria

1. WHEN ユーザーが新しいユーザープロフィールを作成する THEN システムはルートアカウントに紐づけて保存する
2. WHEN ユーザーがプロフィール名を設定する THEN システムは正式名（[タイムスタンプ]\_[名前]）と略称（[名前]）を保存する
3. WHEN ユーザーがプロフィール名を変更する THEN システムはポイントを消費して名前を更新する
4. WHEN ユーザーがプロフィールを削除する THEN システムはそのプロフィールを無効化するが、データは保持する

### Requirement 6

**User Story:** ユーザーとして、グループとアライアンスを管理したいので、グループ管理機能が必要です。

#### Acceptance Criteria

1. WHEN ユーザーが新しいグループを作成する THEN システムはルートアカウントをリーダーとして設定する
2. WHEN ユーザーがグループのルールを設定する THEN システムはオアシス宣言をデフォルトで適用し、追加ルールを保存する
3. WHEN ユーザーがグループを削除する THEN システムはユーザーがリーダーであることを確認してから削除する
4. WHEN ユーザーがアライアンスを作成・管理する THEN システムは同様のルールと権限チェックを適用する

### Requirement 7

**User Story:** ユーザーとして、信頼度システムを利用したいので、信頼度管理機能が必要です。

#### Acceptance Criteria

1. WHEN ルートアカウントが作成される THEN システムは初期信頼度ポイントとして 30 ポイントを設定する
2. WHEN 警告なしで 1 日が経過する THEN システムは信頼度ポイントを 1 ポイント増加させる
3. WHEN ユーザーが警告を受ける THEN システムは warning_count を増加させ last_warning_at を更新する
4. WHEN 信頼度の高いユーザーから警告が発生する THEN システムは信頼度ポイントを 0 にリセットする
5. WHEN ペナルティが適用される THEN システムは Penalties テーブルに記録を追加する

### Requirement 8

**User Story:** ユーザーとして、アカウント情報を管理したいので、編集・リセット機能が必要です。

#### Acceptance Criteria

1. WHEN ユーザーがルートアカウント情報を編集する THEN システムは ID を除く基本情報の変更を許可する
2. WHEN ユーザーがアカウントをリセットする THEN システムは認証データ以外のすべてのデータを初期値に戻す
3. WHEN リセットが実行される THEN システムは累積データをゼロに戻す
4. WHEN ユーザーが状態を更新する THEN システムは複数のユーザープロフィールの最新状態を一元管理する

### Requirement 9

**User Story:** ユーザーとして、ルートアカウントの情報を一元的に管理したいので、専用の管理ページが必要です。

#### Acceptance Criteria

1. WHEN ユーザーがルートアカウント管理ページにアクセスする THEN システムはルートアカウントの基本情報を表示する
2. WHEN ユーザーが管理ページで情報を編集する THEN システムは変更を検証し保存する
3. WHEN ユーザーが管理ページでユーザープロフィール一覧を表示する THEN システムは作成済みのすべてのプロフィールを表示する
4. WHEN ユーザーが管理ページでグループ・アライアンス一覧を表示する THEN システムはユーザーがリーダーのグループとアライアンスを表示する
5. WHEN ユーザーが管理ページでポイント履歴を表示する THEN システムはポイントの獲得・消費履歴を表示する
6. WHEN ユーザーが管理ページで信頼度情報を表示する THEN システムは現在の信頼度ポイントと履歴を表示する
