# OAuth認証フロー - ログイン/初回登録仕様

> **作成日**: 2026-02-28
> **更新日**: 2026-02-28
> **目的**: Google/GitHub OAuth認証を活用したログイン・初回登録フロー定義
> **要件**: メール/パスワード認証は禁止。OAuth（Google/GitHub）のみを使用

---

## 📋 概要

このドキュメントは、**Google/GitHub OAuth認証を活用したアカウント作成・ログインフロー** を定義します。
メール/パスワード認証は使用禁止で、OAuth プロバイダーのみでユーザー管理を行います。

---

## 🎯 基本方針

1. **OAuth のみ**: Google/GitHub 認証を唯一のログイン手段
2. **シンプルなフロー**: ボタンクリック → プロバイダー認証 → 初回登録（表示名） → ホーム
3. **段階的な情報収集**: 初回登録時に最小限の必須情報のみ取得
4. **自動ユーザー作成**: Better Auth が OAuth データからユーザーを自動作成

---

## 🔑 ログイン・初回登録フロー

### 1. ログインページ（src/app/(public)/login/page.tsx）

#### 表示要素

| 要素 | 説明 |
|------|------|
| ページタイトル | 「ログイン」または「アカウントにサインイン」 |
| Google認証ボタン | 既存実装を活用 |
| GitHub認証ボタン | 既存実装を活用 |
| サポートリンク | FAQ/ヘルプ（任意） |

#### UIフロー

```
1. ログインページ（/login）にアクセス
   ├─ 「Googleでサインイン」ボタン
   └─ 「GitHubでサインイン」ボタン

2. ボタンクリック
   ├─ OAuth プロバイダーのページへリダイレクト
   ├─ ユーザーが認証
   └─ Better Auth → アプリケーションへ戻る

3. 初回ユーザーの場合
   ├─ 初回登録フォームへ（表示名入力）
   └─ データ保存後、ホームへリダイレクト

4. 既存ユーザーの場合
   └─ セッション開始、ホーム（/home）へリダイレクト
```

### 2. OAuth後の初回登録フォーム

#### 必須項目

| 項目 | 説明 | バリデーション |
|------|------|----------------|
| 表示名 | ユーザーの表示名（OAuth から自動取得、編集可能） | 3文字以上、50文字以下 |

#### 自動取得項目（Better Auth が処理）

| 項目 | 説明 | 取得元 |
|------|------|--------|
| メールアドレス | OAuth プロバイダーから自動取得（DBに保存） | Google/GitHub |
| プロフィール画像 | OAuth プロバイダーから自動取得（任意） | Google/GitHub |

#### UIフロー
| プロフィール画像 | OAuthプロバイダーから自動取得 | Google/GitHub |

#### UIフロー

```
1. ログイン/サインアップページにアクセス
   ├─ 「Googleでログイン」ボタン
   └─ 「GitHubでログイン」ボタン

2. OAuthボタンクリック
   ├─ OAuthプロバイダーの認証画面へリダイレクト
   ├─ ユーザーが認証を許可
   └─ Better Auth がユーザー情報を取得

3. 初回登録の場合
   ├─ 表示名入力ダイアログを表示（自動取得された名前が初期値）
   ├─ ユーザーが表示名を確認・編集
   └─ 「登録」ボタンクリック → /home へリダイレクト

4. 既存ユーザーの場合
   └─ そのまま /home へリダイレクト
```

## 🎭 お試し体験（`anonymous`）の扱い

### 入力項目

- **入力不要**: お試し体験では一切の入力を求めない
- **LocalStorageベース**: すべてのデータはLocalStorageに保存
- **制限モード**: DB書き込み不可、Server Action一部のみ実行可能

### UIフロー

```
1. ランディングページ（/）にアクセス
   ├─ 「お試しで始める」ボタンを表示
   └─ 「ログイン」「サインアップ」リンクも表示

2. 「お試しで始める」ボタンクリック
   ├─ Better Auth の anonymous ログインを実行
   ├─ LocalStorageに `anonymous` フラグを保存
   └─ /home へリダイレクト（制限モード）

3. お試しホーム（/home）
   ├─ 「本登録へ移行」ボタンを表示
   ├─ 「お試しデータを削除」ボタンを表示
   └─ LocalStorageからデータを読み込んで表示
```

## 🔄 お試し→本登録移行時の動作

### 移行ダイアログの表示内容

**メッセージ**: 「本登録に進みますか？お試しデータは削除されます」

### UIフロー

```
1. お試しホーム（/home-trial）で「本登録へ進む」ボタンクリック
   └─ 移行確認ダイアログを表示

2. 移行確認ダイアログ
   ├─ メッセージ表示: 「本登録に進みますか？」
   ├─ 「キャンセル」ボタン（ダイアログを閉じる）
   └─ 認証方法を選択するボタン

3. OAuth選択の場合
   ├─ 「Googleで登録」または「GitHubで登録」ボタンをクリック
   ├─ OAuthプロバイダーの認証画面へリダイレクト
   ├─ ユーザーが認証を許可
   ├─ Better Auth がユーザー情報を取得し、新規アカウントを作成
   ├─ クライアント側で `clearTrialData()` を実行（LocalStorageをクリア）
   └─ /home へリダイレクト（本登録ユーザーとして）

4. ⚠️ 重要なポイント
   ├─ trial データはサーバーに保存されない
   ├─ trial データはサーバーに移行されない
   ├─ OAuth ログイン後、新規アカウントが作成される（trial とは無関係）
   └─ LocalStorage 内のデータのみが削除される
```

## 🔧 バリデーションルール

### 表示名

- **最小長**: 3文字
- **最大長**: 50文字
- **許可文字**: 英数字、日本語、スペース、一部記号（`-`, `_`, `.`）
- **禁止文字**: HTML特殊文字（`<`, `>`, `&`, `"`）

```typescript
export function validateDisplayName(name: string): boolean {
  const trimmed = name.trim();
  if (trimmed.length < 3 || trimmed.length > 50) return false;
  if (/<|>|&|"/.test(trimmed)) return false; // HTML特殊文字を禁止
  return true;
}
```

### メールアドレス

- **形式**: Email形式（RFC 5322準拠）
- **最大長**: 254文字
- **重複チェック**: DB上で重複しないことを確認

```typescript
export function validateEmail(email: string): boolean {
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  return emailRegex.test(email) && email.length <= 254;
}
```

### パスワード

- **最小長**: 8文字
- **最大長**: 128文字
- **必須要素**: 英字・数字・記号のうち2種類以上
- **禁止**: 平文パスワードの保存（Better Authが自動でハッシュ化）

```typescript
export function validatePassword(password: string): boolean {
  if (password.length < 8 || password.length > 128) return false;

  const hasLetter = /[a-zA-Z]/.test(password);
  const hasNumber = /[0-9]/.test(password);
  const hasSymbol = /[!@#$%^&*()_+\-=\[\]{};':"\\|,.<>\/?]/.test(password);

  // 2種類以上必須
  return [hasLetter, hasNumber, hasSymbol].filter(Boolean).length >= 2;
}
```

## 💡 Tips

### OAuth優先の理由

- **パスワードレス**: ユーザーがパスワードを管理する必要がない
- **セキュリティ**: OAuthプロバイダーのセキュリティ対策を活用
- **利便性**: ワンクリックでログイン可能

### お試し体験の低摩擦化

- **入力不要**: 即座に体験開始（コンバージョン率向上）
- **LocalStorageベース**: サーバーサイドの負荷を軽減
- **段階的な移行**: 本登録への移行をスムーズに

### バリデーションのタイミング

- **クライアント側**: リアルタイムフィードバック（UX向上）
- **サーバー側**: 最終的な検証（セキュリティ担保）
- **二重チェック**: クライアント・サーバーの両方で実施

---

**GitHub Copilot より**: アカウント入力の最小化は、ユーザー体験を向上させる重要な要素です。OAuth優先戦略により、パスワード管理の負担を軽減し、セキュリティも向上します。また、お試し体験の低摩擦化により、コンバージョン率の向上が期待できます。
