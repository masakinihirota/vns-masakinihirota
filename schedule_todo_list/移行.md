## 🎯 推奨選択肢（推奨度順）

### **Option A: 完全なバレルインポート階層構造（推奨度: ★★★★★ 95%）**

**構造:**
```
src/components/landing-page/
├── index.ts                            # バレル: export * from "./tales-claire"
├── tales-claire/
│   ├── index.ts                       # バレル: 全セクションをexport
│   ├── tales-claire-lp.tsx            # メインコンポーネント
│   ├── sections/                      # セクション専用フォルダ
│   │   ├── hero-section.tsx
│   │   ├── purpose-section.tsx
│   │   ├── inspiration-section.tsx
│   │   └── ...
│   └── components/                    # tales-claire専用コンポーネント
│       ├── background-canvas.tsx
│       └── local-mode-button.tsx

    ├── index.ts
    ├── trial-entry/
    └── identity-section/

src/app/(public)/(static)/tales-claire/page.tsx:
import { TalesClaireLP } from "@/components/landing-page";
```

**メリット:**
- ✅ **最もクリーン**: 階層が明確で、将来的に複数のLPを追加しやすい
- ✅ **保守性最高**: sections/とcomponents/で責務が分離
- ✅ **バレルインポート完全準拠**: vns既存設計と100%一致
- ✅ **共有部分の明確化**: `shared/` フォルダで他LPとの共有が容易
- ✅ **テスト配置が自然**: `sections/hero-section.test.tsx` のようにコロケーション

**デメリット:**
- ⚠️ フォルダ階層が深くなる（3~4層）
- ⚠️ 移行時のファイル配置作業がやや複雑

**推奨理由:**
- antiの設計思想を完全に踏襲しつつ、vns-masakinihirotaの既存パターンと統合可能
- 将来的に `/tales-claire-v2` や `/another-lp` を追加する際の拡張性が最も高い

- **バレルインポートのパフォーマンス**: Next.js 16のTurbopackは tree-shaking が優秀なため、バレルインポートによるバンドルサイズ増加は最小限です。
