## 🎯 修正方針の選択肢

### Option A: 全面修正（完全版）⭐⭐⭐⭐⭐ **【最推奨】**

**内容:**
- ✅ 優先度 1（Critical）の 3 項目を全て修正
- ✅ 優先度 2（Major）の 4 項目を全て修正

**メリット:**
- ✅ レビュー指摘事項を完全に解決
- ✅ 実装開始前にリスクを最小化
- ✅ ステークホルダーへの説明責任を果たせる
- ✅ 後続の実装判断が容易になる
- ✅ プロジェクト失敗のリスクが大幅に低減

**デメリット:**
- ⚠️ 修正に 1.5～2h かかる
- ⚠️ 一部の判断（Server Actions 方針など）で追加の意思決定が必要

**推奨度:** ⭐⭐⭐⭐⭐ **100%**

**理由:**
1. **現在のマスタープランは「実装不可能」**
   - 「決定済み」と主張しているが PoC 未実施

2. **1.5～2h の投資で 10～20h の無駄を防げる**

   - これを防ぐための「保険」として、修正コストは妥当

3. **PoC 前に完璧な計画が必要**
   - PoC は「計画の妥当性を検証」するためのもの
   - 計画が不完全なまま PoC を実施すると、検証すべきポイントが不明確

### フェーズごとのロールバック判断基準

**フェーズ 2（Admin API）:**
- 🔴 RBAC が正常動作しない → **Server Actions に戻す、Hono は見送り**
- 🟠 既存テスト改修が困難 → **Server Actions は維持、Admin API のみ Hono**

### ロールバック手順
1. **即座に判断:** エラーが Critical（🔴）か Major（🟠）かを判断
2. **git revert:** 該当コミットを取り消し
3. **動作確認:** `pnpm build && pnpm test` で既存機能の動作確認
4. **Staging テスト:** デプロイ前に Staging で動作確認
5. **ポストモーテム:** 失敗原因を分析し、`.agent/postmortem/` に記録
