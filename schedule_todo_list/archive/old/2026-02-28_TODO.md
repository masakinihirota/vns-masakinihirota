## 💡 Tips

### TDD（テスト駆動開発）のサイクル
1. **Red**: テストを書いて失敗させる
2. **Green**: 最小限の実装でテストを通す
3. **Refactor**: コードを改善する

### Deny-by-defaultの原則
- すべての権限チェックは「明示的に許可されない限り拒否」が基本
- Server Actionの冒頭で必ず権限チェックを実行
- UIガードとServer Actionガードの両方を実装（二重防御）

### 組織と国の分離
- 組織（Group）: ボトムアップ、同じ価値観、実行単位
- 国（Nation）: トップダウン、複数組織、ネットワーキング
- 別エンティティとして管理し、権限・フロー・状態遷移を分離

### Next.js 16 対応
- `proxy.ts` を使用（`middleware.ts` は非推奨）
- `await params`, `await searchParams` 必須（Async Request APIs）
- Server Actions を優先（Route Handlers より型安全）

### Better Auth 運用
- `pnpm db:auth:check` でスキーマ整合性確認
- `pnpm db:auth:fix-compat` で不整合修復
- 環境変数の確認（`.env.local`）

### テスト配置方針
- システムテスト: `src/__tests__/` または `src/lib/**/__tests__/`
- コンポーネントテスト: コロケーション（`page.test.tsx` を同階層に配置）

---

**GitHub Copilot より**: RBAC実装では、権限マトリクスが実装とテストの両方の「信頼できる唯一の情報源（SSOT）」になります。まずマトリクスを固め、それからテストを書き、最後に実装することで、仕様の曖昧さを排除できます。また、`RoleType` と `RelationshipType` を型レベルで分離することで、TypeScriptの恩恵を最大限に受けられます。
