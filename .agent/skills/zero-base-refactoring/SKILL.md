# Zero-Base Refactoring (再構築の勇気)

このスキルは、スパゲッティコードや場当たり的な修正が積み重なった際、あるいは不適切な初期設計が判明した際に、過去のコードへの執着を捨てて「あるべき姿」をゼロから定義し、再構築するための行動指針です。

## 1. 再構築の勇気 (The Courage to Rebuild)

- **サンクコストを無視する**: これまで費やした時間や労力に囚われず、現状のコードが「エレガントでない」かつ「負債になっている」と判断した場合、迷わず「捨てる」決断をします。
- **エレガントな解決策の追求**: 単に書き直すのではなく、現在の文脈、技術スタック、およびこれまでに得た教訓をすべて踏まえ、最も洗練されたシンプルな設計（エレガントな解決策）を実装します。

## 2. ゼロベース評価基準 (Decision Criteria)

以下の兆候が見られた場合、(`/zerobase`) プロトコルの発動を検討または提案します：

- **カオスな修正の連鎖**: 1つのバグを直すと別のバグが出る状態。
- **理解の限界**: コードの意図が不明瞭で、リファクタリングするよりも一から書いたほうが早いと判断される場合。
- **技術的負債の閾値超過**: 既存の構造が新しい要件を阻害し、拡張性が完全に失われている場合。
- **「継ぎ接ぎ」の限界**: `if` 文やフラグが乱立し、本来のロジックが埋もれてしまっている場合。

## 3. 再構築プロセス (Reconstruction Process)

再構築を行う際は、以下のステップを遵守します：

1.  **ゴール（あるべき姿）の定義**: 「もし今、白紙の状態から始めるならどう書くか」を極限まで突き詰めます。
2.  **既存仕様の抽象化**: 捨てるコードから「守るべきビジネスロジック」と「重要なエッジケース」だけを抽出し、ドキュメント化します。
3.  **クリーンな再実装**:
    - 最新のベストプラクティスを適用します。
    - 徹底したTDDにより、以前のコードよりも高い堅牢性を保証します。
4.  **移行と検証**: 以前の機能を損なうことなく、より高性能・高品質な解決策に置き換わったことを証明します。

## 4. ユーザーとの合意

「これを捨てて、やり直します」と提案する際は、その理由（なぜ捨てるべきか）と、再構築によって得られるメリット（保守性、パフォーマンス、可読性の向上）を明確に説明します。

**目標**: 過去の自分を超え、常に「今できる最高にエレガントなコード」をプロジェクトに提供し続けます。
