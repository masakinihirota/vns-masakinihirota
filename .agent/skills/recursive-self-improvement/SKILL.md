# Recursive Self-Improvement (再帰的な自己改善)

このスキルは、発生したミスを単なる「エラー」として片付けず、将来の自分（Antigravity）が同じ間違いを繰り返さないための「知識」として昇華させるための行動指針です。

## 1. 再帰的ループの原則 (The Recursive Loop)

- **ミスは学習の機会**: 予期せぬビルドエラー、ユーザーからの指摘、バグの発生、または非効率な手順を経験した際、即座に「なぜこれが起きたか」を分析します。
- **ルールの永続化**: ミスの原因が「不備のあるルール」または「ルールの欠如」にある場合、即座に関連する設定ファイル（`.agent/skills/*.md`, workflows, またはユーザー提供のルール）を更新します。

## 2. 根本原因分析 (Root Cause Analysis - RCA)

ミスが発生した際、以下のステップを実行します：

1.  **直接原因の特定**: どのコード、どのコマンドが直接的なエラーを引き起こしたか。
2.  **背後にある知識不足の発見**: 私が何を知らなかったか、または何を誤解していたか。
3.  **プロセスの欠陥の特定**: 既存のワークフローで、なぜこのミスを事前に防げなかったか。
4.  **改善策の策定**: 二度と繰り返さないために、どのドキュメントにどんな制約やヒントを追加すべきか。

## 3. 自己改善の実行手順

ミスを修正・リカバリーする際、以下の行動を必ず含めます：

- [ ] **ミスの記録**: `decision_log.md` 等に発生した事象と原因を簡潔に記録。
- [ ] **ルールのブラッシュアップ**:
  - `SKILL.md`: スキルの制約やチェックリストを強化します。
  - `workflows`: 手順に「確認ステップ」を自動追加します。
  - `user_global` 等のメモリ: ユーザー体験に直結する教則を追記します。
- [ ] **改善の証明**: 更新されたルールに基づき、再度同じ操作を行ってもミスが起きないことを確認（または思考実験）します。

## 4. ユーザーへの報告

改善を実施した際は、「このミスを二度としないよう、〇〇のルールを更新しました」とユーザーに明示的に報告し、改善のサイクルを可視化します。

**目標**: 失敗を重ねるごとに「より賢く、より正確で、より信頼できるエージェント」へと指数関数的に進化し続けます。

## 5. 学習履歴 (Learning Log)

### 2026-02-27: Next.js 16 middleware → proxy 誤判断事件

**発生した問題**:
- ビルドエラー「Invalid segment configuration export detected」が発生
- `src/proxy.ts` が原因と特定し、一時的に削除してビルドが進んだ
- エージェントは「proxy.ts は誤った設定」と判断し、削除・改名を提案

**ユーザーの指摘**:
> 「proxy.ts はNext.js 16の新しい設定です、確認してください」

**根本原因**:
1. **知識の陳腐化**: Next.js 16で `middleware.ts` が `proxy.ts` に公式改名されたことを知らなかった
2. **仮定の誤り**: ビルドエラーとファイル存在の相関から「proxy.ts が問題」と早合点した
3. **確認不足**: 公式ドキュメントを確認せず、推測で判断した

**真の原因**:
- `searchParams` が Next.js 16 で `Promise` 型に変更されたことへの未対応
- UUID 型カラムへの `ilike` 演算子使用（PostgreSQL の型制約違反）

**学んだ教訓**:
1. ✅ **メジャーバージョンアップ時は必ず公式ドキュメントを確認する**
   - Next.js 16 の変更点: https://nextjs.org/blog/next-16
   - API Reference: https://nextjs.org/docs/app/api-reference/file-conventions/proxy

2. ✅ **ユーザーの指摘を優先する**
   - エージェントの推測 < ユーザーの知識
   - 疑問があれば即座にドキュメント確認

3. ✅ **相関関係と因果関係を混同しない**
   - proxy.ts を削除 → ビルドが進んだ ≠ proxy.ts が原因
   - 別のエラーが露呈しただけ

4. ✅ **Next.js 16 の重要な変更点**
   - `middleware.ts` → `proxy.ts` (公式改名)
   - `params`, `searchParams` が `Promise<T>` 型に変更
   - `await params`, `await searchParams` が必須

**実装した改善**:
- `searchParams: Promise<{...}>` に型を修正
- UUID 型カラム検索時に `sql`\`\${column}::text ilike ...\` でキャスト
- proxy.ts の matcher パターンを標準形式に修正

**今後の行動指針**:
- フレームワークのメジャーバージョンアップ時は Breaking Changes を最優先で確認
- ビルドエラー発生時は「削除して動くか」ではなく「なぜエラーが出るか」を分析
- ユーザーが「これは正しい」と主張した場合、まず公式情報を確認してから判断
